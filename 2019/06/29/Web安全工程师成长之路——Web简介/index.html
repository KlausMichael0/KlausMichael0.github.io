Web安全工程师之路——第一章Web简介

Posted on 2019-06-29 | Edited on 2019-07-10

Web简介，从入门到入狱

第一章–Web简介

第一节–Web介绍

  • Web—World Wide Web（万维网）

    • Web发展史
      • Web1.0（服务器端）——Web2.0（客户端）
    • Web工作流程
      1. 用户在浏览器中输入网址。
      2. 浏览器向对应网址的服务器发送请求。
      3. 服务器与数据库交互。
      4. 服务器向浏览器返回请求结果。
      5. 浏览器向用户返回响应界面。
    • 浏览器的工作原理
      1. 浏览器通过域名获取Web服务器IP地址（DNS解析）
        • 浏览器向DNS服务器发送DNS请求。
        • DNS服务器发送DNS响应。
      2. 浏览器根据IP地址与Web服务器通过HTTP协议交互
        • 浏览器向Web服务器发送HTTP请求。
        • Web服务器发送HTTP响应。
      3. 浏览器返回给用户响应界面

  • Web架构的熟悉

    • 常见的Web前端相关的开发语言

      • HTML—超文本标记语言。
      • CSS—层叠样式表。
      • JavaScript—直译式脚本语言。

    • 常见的Web后端相关的开发语言

      • Java—跨平台的语言，有许多优秀的框架。
      • PHP—解释执行的脚本语言。
      • GO—用Google公司开发的一门语言。

    • 服务器（Server）

      1. 有时Server表示硬件，也就是一台机器，它还有另一个名字：[主机]。
      2. 更多时候，Server表示软件程序，这种程序主要用来对外提供某些服务，比如邮件服务、FTP服务、数据库服务、网页服务等。
      • 一般作为开发者谈及Server时指的是后者，也就是一个24小时运行的软件程序，一台主机上可以运行多个这样的程序。

    • 数据库服务器

      • 由运行在局域网中的一台/多台计算机和数据库管理系统软件共同构成，数据库服务器为客户应用程序提供数据服务。

    • Web服务器

      • 提供Web服务的服务器（一般对外提供的是HTTP服务，也可以是其他服务）。

    • DNS服务器

      • 进行域名和与之相对应的IP地址转换的服务器。
      • DNS中保存了一张域名和与之对应的IP地址的表，以解析消息的域名。

  • Question：我们在浏览器输入URL以后的Web流程

    1. DNS解析
    2. 建立连接
    3. 发送HTTP请求
    4. 返回HTTP响应
    5. 维持连接
    6. 断开连接
    7. 浏览器解析

    具体的过程在谢老师的计网书中有详细讲到。

第二节–Web通信

  • URL是什么、作用

    • URL是统一资源定位符，是因特网上标准的资源的地址。
    • 通过URL可以到达任何一个地方寻找需要的东西，比如文件、数据库、图像、新闻组等等，可以这样说，URL是Internet上的地址簿。

  • URL格式

    • scheme：使用的底层协议。

    • authority：服务器的域名或IP地址+服务器端口（HTTP默认端口为80）。

    • path：访问资源的路径。

    • query：发送给http服务器的数据。

    • fragment：锚，用来指导浏览器动作的，对于服务器端完全无用。

  • HTTP的报文结构

    • HTTP请求
      • 请求行（字段之间使用空格分割）
        • 请求方法字段
          • GET方法：要求服务器将URL定位的资源放在响应报文的数据部分，回送给客户端，传递参数长度受限制，请求数据以地址的形式表现在请求行。
          • POST方法：允许客户端给服务器提供较多的信息，将请求参数封装在HTTP请求数据中，大多用在页面的表单中。
        • URL字段
        • HTTP协议版本字段
      • 请求头部——由关键字/值对组成，每行一对，关键字和值用英文冒号:分割
      • 空行
      • 请求数据
        • 不在GET方法中使用，而是在POST方法中使用。
    • HTTP响应
      • 状态行——HTTP-Version Status-Code Reason-Phrase CRLF
        • HTTP-Version表示服务器HTTP协议的版本。
        • Status-Code表示服务器发回的响应状态代码。
        • Reason-Phrase表示状态代码的文本描述。
        • CRLF为回车换行。
      • 消息报头
      • 响应正文

  • Cookies和Set-Cookie

    • Web服务器通过发送一个Set-Cookie的HTTP信息头来创建一个cookies。
    • 当存在一个 cookie，并允许设置可选项，该 cookie 的值会在随后的每次请求中被发送至服务器，cookie 的值被存储在名为 Cookie 的 HTTP 消息头中，并且只包含了 cookie 的值，忽略全部设置选项。

  • Referer（HTTP请求头的一部分）的作用

    • 指示一个请求是从哪里链接过来，那么当一个请求并不是由链接触发产生的，那么自然也就不需要指定这个请求的链接来源。
    • 防盗链——只允许自己的网站访问自己的图片服务器。
    • 防止恶意请求——所有的动态请求必须Referer为自己的网站。

  • 状态码

    • 1xx：指示信息–表示请求已接收，继续处理。
    • 2xx：成功–表示请求已被成功接收、理解、接受。
    • 3xx：重定向–要完成请求必须进行更进一步的操作。
    • 4xx：客户端错误–请求有语法错误或请求无法实现。
    • 5xx：服务器端错误–服务器未能实现合法的请求。

  • 常见的状态代码、状态描述

    • 200 OK：客户端请求成功。
    • 400 Bad Request：客户端请求有语法错误，不能被服务器所理解。
    • 401 Unauthorized：请求未经授权，这个状态代码必须和WWW-Authenticate报头域一起使用。
    • 403 Forbidden：服务器收到请求，但是拒绝提供服务。
    • 404 Not Found：请求资源不存在，举个例子：输入了错误的URL。
    • 500 Internal Server Error：服务器发生不可预期的错误。
    • 503 Server Unavailable：服务器当前不能处理客户端的请求，一段时间后可能恢复正常，举个例子：HTTP/1.1 200 OK（CRLF）。

  • Location

    • 要求网页浏览器加载其他网页，使用HTTP状态码3xx发送Location头（Location header）。
    • 提供有关新创建资源位置的信息。在这种情况下应该使用HTTP状态码201或202发送Location头。
  • Q1：熟悉常见浏览器协议
    • ftp/http/file，列出每一种协议的具体说明
      • ftp：FTP（File Transfer Protocol，文件传输协议） 是 TCP/IP 协议组中的协议之一。FTP协议包括两个组成部分，其一为FTP服务器，其二为FTP客户端。其中FTP服务器用来存储文件，用户可以使用FTP客户端通过FTP协议访问位于FTP服务器上的资源。在开发网站的时候，通常利用FTP协议把网页或程序传到Web服务器上。此外，由于FTP传输效率非常高，在网络上传输大的文件时，一般也采用该协议。默认情况下FTP协议使用TCP端口中的 20和21这两个端口，其中20用于传输数据，21用于传输控制信息。但是，是否使用20作为传输数据的端口与FTP使用的传输模式有关，如果采用主动模式，那么数据传输端口就是20；如果采用被动模式，则具体最终使用哪个端口要服务器端和客户端协商决定。
      • http：HTTP（Hyper Text Transfer Protocol，超文本传输协议）是用于从万维网（WWW:World Wide Web ）服务器传输超文本到本地浏览器的传送协议。基于TCP/IP通信协议来传递数据（HTML 文件, 图片文件, 查询结果等）。HTTP协议工作于客户端-服务端架构为上。浏览器作为HTTP客户端通过URL向HTTP服务端即WEB服务器发送所有请求。Web服务器根据接收到的请求后，向客户端发送响应信息。
      • file：FILE（File Protocol，本地文件传输协议）主要用于访问本地计算机中的文件，就如同在Windows资源管理器中打开文件一样。
    • 使用这3种类协议访问资源，了解访问结果（包括协议类型和资源）
  • Q2：熟悉HTTP
    • 熟悉HTTP报文
      • 学习HTTP请求报文中的请求方式：GET、POST、HEAD、PUT、DELETE、MOVE、OPTIONS、TRACE的意义，给出总结列表及用法
        • GET：会显示请求的指定资源，一般只用于数据的读取。
        • POST：会向指定资源提交数据，请求服务器进行处理，如：表单提交、文件上传等。
        • HEAD：与GET方法一样，都是向服务器发出指定资源的请求，但是，服务器在响应HEAD请求时不会回传资源的内容部分，即：响应主体。这样，我们可以不传输全部内容的情况下，就可以获取服务器的响应头信息。HEAD方法常被用于客户端查看服务器的性能。
        • PUT：会身向指定资源位置上传其最新内容，PUT方法是幂等(即f(f(x))=f(x))的方法。通过该方法客户端可以将指定资源的最新数据传送给服务器取代指定的资源的内容。
        • DELETE：用于请求服务器删除所请求URI（统一资源标识符，Uniform Resource Identifier）所标识的资源。DELETE请求后指定资源会被删除，DELETE方法也是幂等的。
        • MOVE：请求服务器将指定的页面移至另一个网络地址。
        • OPTIONS：与HEAD类似，一般也是用于客户端查看服务器的性能。 这个方法会请求服务器返回该资源所支持的所有HTTP请求方法，该方法会用’*’来代替资源名称，向服务器发送OPTIONS请求，可以测试服务器功能是否正常。
        • TRACE：请求服务器回显其收到的请求信息，该方法主要用于HTTP请求的测试或诊断。
      • 了解返回报文中常见状态码1XX–5XX的大致含义
        • 1xx：指示信息–表示请求已接收，继续处理。
        • 2xx：成功–表示请求已被成功接收、理解、接受。
        • 3xx：重定向–要完成请求必须进行更进一步的操作。
        • 4xx：客户端错误–请求有语法错误或请求无法实现。
        • 5xx：服务器端错误–服务器未能实现合法的请求。
      • 了解请求报文中User-Agent、Host、Cookie、X-Forwarded-For和返回包中server、Set-Cookie、 Content-Type、httponly、X-Frame-Options属性的含义
        • 请求报文
          • User-Agent：产生请求的浏览器类型。
          • Host：请求的主机名，允许多个域名同处一个IP地址，即虚拟主机。
          • Cookie：HTTP请求发送时，会把保存在该请求域名下的所有cookie值一起发送给web服务器。
          • X-Forwarded-For：用来识别通过HTTP代理或负载均衡方式连接到Web服务器的客户端最原始的IP地址的HTTP请求头字段。
        • 响应报文
          • server：Web服务器软件名称。
          • Set-Cookie：设置http Cookie。
          • Content-Type：返回内容的MIME类型（多用途互联网邮件扩展类型，是设定某种扩展名的文件用一种应用程序来打开的方式类型，当该扩展名文件被访问的时候，浏览器会自动使用指定应用程序来打开）。
          • httponly：当cookie 有设定HttpOnly flag 时，浏览器会限制cookie 只能经由HTTP(S) 协定来存取。因此当网站有XSS 弱点时，若cookie 含有HttpOnly flag，则攻击者无法直接经由JavaScript 存取使用者的session cookie，可降低使用者身份被盗用的机率。
          • X-Frame-Options：用来给浏览器指示允许一个页面可否在<frame>、<iframe>或者<object>中展现的标记。网站可以使用此功能，来确保自己网站的内容没有被嵌到别人的网站中去，也从而避免了点击劫持 （clickjacking） 的攻击。
    • 了解Base64编码（通常以一个或两个=结尾）
      • 一种基于64个可打印字符来表示二进制数据的表示方法，3个字节可由4个可打印字符来表示。
      • 如果要编码的字节数不能被3整除，最后会多出1个或2个字节，那么可以使用下面的方法进行处理：先使用0字节值在末尾补足，使其能够被3整除，然后再进行Base64的编码。在编码后的Base64文本后加上一个或两个=号，代表补足的字节数。也就是说，当最后剩余两个八位字节（2个byte）时，最后一个6位的Base64字节块有四位是0值，最后附加上两个=号；如果最后剩余一个八位字节（1个byte）时，最后一个6位的base字节块有两位是0值，最后附加一个=号。
      • Base64常用于在通常处理文本数据的场合，表示、传输、存储一些二进制数据，包括MIME的电子邮件及XML（可扩展标记语言，通过此种标记，计算机之间可以处理包含各种信息的文章等）的一些复杂数据。
    • 熟悉HTTP基本认证
      • 概念
        • 基本认证（Basic access authentication）是一种用来允许网页浏览器或其他客户端程序在请求时提供用户名和口令形式的身份凭证的一种登录验证方式。
        • 在发送之前是以用户名追加一个:号然后串接上口令，并将得出的结果字符串再用Base64算法编码。例如，提供的用户名是Aladdin、口令是open sesame，则拼接后的结果就是Aladdin:open sesame，然后再将其用Base64编码，得到QWxhZGRpbjpvcGVuIHNlc2FtZQ==。最终将Base64编码的字符串发送出去，由接收者解码得到一个由冒号分隔的用户名和口令的字符串。
        • 编码这一步骤的目的并不是安全与隐私，而是为将用户名和口令中的不兼容的字符转换为均与HTTP协议兼容的字符集。
      • 优点
        • 基本所有流行的网页浏览器都支持基本认证
      • 缺点
        • 该方案建立在以下的假设的基础上，即：客户端和服务器主机之间的连接是安全可信的。特别是，如果没有使用SSL/TLS这样的传输层安全的协议，那么以明文传输的密钥和口令很容易被兰姐。
        • 现存的浏览器保存认证信息直到标签页或浏览器被关闭，或者用户清除历史记录。这意味着服务器端在用户不关闭浏览器的情况下，并没有一种有效的方法来让用户登出。
  • Q3（high-level）：简述HTTPS和HTTP的区别和联系，了解跟HTTPS相关的CA证书、SSL的概念
    • 简单来说，HTTPS协议是由SSL+HTTP协议构建的可进行加密传输、身份认证的网络协议，要比http协议安全。
    • 详细解析HTTP与HTTPS的区别
    • CA证书
      • 如果用户想得到一份属于自己的证书，他应先向 CA 提出申请。在 CA 判明申请者的身份后，便为他分配一个公钥，并且 CA 将该公钥与申请者的身份信息绑在一起，并为之签字后，便形成证书发给申请者。
      • 如果一个用户想鉴别另一个证书的真伪，他就用 CA 的公钥对那个证书上的签字进行验证，一旦验证通过，该证书就被认为是有效的。证书实际是由证书签证机关（CA）签发的对用户的公钥的认证。
    • SSL—为网络通信提供安全及数据完整性的一种安全协议
      • 认证用户和服务器，确保数据发送到正确的客户机和服务器。
      • 加密数据以防止数据中途被窃取。
      • 维护数据的完整性，确保数据在传输过程中不被改变。
